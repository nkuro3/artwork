# /spec — 対話的に仕様書を作成

`$ARGUMENTS` = 要件概要（このコマンドで作る仕様の出発点）。空なら、まず「何の仕様を作るか」を尋ねること。

このコマンドの責務は **仕様書1本を作ること**のみ。実装・タスク化はしない（タスク化は別途）。決定の*理由*は仕様に書かず ADR 追記を提案する。

## 重要: 対話はメインセッションで行う

仕様詰めはユーザーとの対話が要る。**サブエージェントには委譲しない**（サブエージェントは対話できない）。下調べ（コードベース探索）だけなら Explore エージェントに投げてよい。

## 進め方

### 1. 質問の前に自分で調べる
- `.claude/rules/spec-format.md` を読み、フォーマットに従う。
- 既存 `docs/specs/` を確認 → 次の連番 `NN` と、既存仕様の粒度・スタイルを把握。
- `CLAUDE.md` と `docs/adr/` で前提・既存の技術決定を把握（仕様はそれに矛盾しないように）。
- 概要に関係するコードを軽く調べる（必要なら Explore エージェント）。

### 2. 不足点を対話で詰める（`AskUserQuestion` を使う）
- **一度に全部聞かない**。要点を 1〜数問ずつ、数ラウンドに分ける。各質問は 2〜4 の選択肢 + 自由記入。推奨案には「(推奨)」を付け先頭に。
- 詰める観点は **その仕様に関係するものだけ**: 目的 / 成功条件・想定ユーザー・スコープ(in/out)・主要機能・非機能/制約・データ・画面・セキュリティ。
- 概要やコード・ADR から自明な点は**質問しない**。不明・重要・判断が分岐する点だけ聞く。
- ユーザーの回答で新たな論点が出たら追って確認する。
- **オープン論点を管理する**。会話側でも、ドラフト末尾の「未確認事項（作業用）」節でもよい（詰める足場として便利なら節を置く）。各論点は確定前に必ず終端させる（`spec-format.md` 参照）: ①決定→反映 / ②今回やらない→Out of scope / ③技術判断・未確定→ADR 移管 / ④先送り→別仕様 / ⑤前提を置く（概要内・成功条件の次）。「先送り」も曖昧にせずトリガ/移管先を伴う決定にする。

### 3. ドラフト → レビュー → 承認
- `spec-format.md` に従ってドラフト。**関係しないセクションは出さない**。**テーブル優先**（副項目があるものは表、箇条書きは単純羅列のみ）。
- 全文を提示してレビューを求める。フィードバックで反復。
- **承認ゲート = オープン論点ゼロ**。未終端の論点が残る限り完成にしない。残っていれば対話で終端させる。**Fix 時に「未確認事項（作業用）」節は中身を空にするのではなく節ごと削除する**（`未確認事項: なし` も残さない）。
- **明示的な承認**を得てから保存（spec-driven の承認ゲート / CLAUDE.md）。承認前にファイルへ書かない。

### 4. 保存と後続案内
- `docs/specs/NN-<slug>.md` に保存（`NN` は既存の次番号、`slug` は英小文字ケバブ）。
- 技術選定・アーキの**決定事項があれば**「ADR（`docs/adr/`）に追記しますか？」と提案（仕様には理由を書かない方針のため）。追記は承認後に行う。
- 次段（計画・タスク化）は別作業である旨を一言案内して終了。

## 制約

- 推測で埋めない。不明点は質問する。
- 承認なしに `docs/specs/` へ書き込まない。
- スコープ外（実装・タスク化・ADR本文の確定）に踏み込まない。
