# 仕様書フォーマット（docs/specs/）

`/spec` および手動の仕様作成で従う、確立済みのフォーマット規約。仕様書のテンプレートも兼ねる。

## ファイル規約

| 項目 | ルール |
|---|---|
| 置き場所 | `docs/specs/NN-<slug>.md` |
| NN | 2桁の連番（既存の最大 + 1） |
| slug | 英小文字ケバブケース（例 `02-payments`） |
| 図 | `docs/specs/drawio/` 等に置き、相対リンクで参照 |

## セクション構成

この順で書く。**その仕様に関係しないセクションは丸ごと省く**（無理に全節を埋めない）。

| # | セクション | 内容 | 主な書式 |
|---|---|---|---|
| 1 | 概要 | 何を・なぜ作るか（1〜数段落）＋ 成功条件 | 文 + 成功条件は箇条書き or 番号 |
| 2 | スコープ | In scope / Out of scope | 箇条書き（単純羅列のため可） |
| 3 | 用語 | プロジェクト固有語の定義 | テーブル `用語 | 意味` |
| 4 | 機能要件 | ユーザーが行えること | FR テーブル（下記） |
| 5 | 非機能要件 | 性能・セキュリティ・運用・制約 | NFR テーブル（下記） |
| 6 | データモデル | エンティティのテーブル設計 | エンティティ単位のテーブル（下記） |
| 7 | 画面構成 | 画面/ルートと役割 | テーブル（下記） |

## 書式ルール（重要）

- **テーブル優先**。副項目・説明・属性・状態を持つものは必ずテーブルにする。
- **箇条書きは「単なる羅列」のときだけ**（説明が付くなら表へ）。
- 決定の*理由*は仕様に書かず ADR（`docs/adr/`）へ回す。仕様は「何を」、ADR は「なぜ」。
- 未確定点は `要確認` ラベルを付けるか、末尾「未確認事項」に列挙する。

### 各テーブルの標準列

| 用途 | 列 |
|---|---|
| 機能要件 (FR) | `ID \| 分類 \| 機能要件 \| 備考`（ID は `FR-01`…） |
| 非機能要件 (NFR) | `ID \| 分類 \| 要件・方針 \| 備考`（`NFR-01`…。セキュリティは `SEC-01`… として節を分けてもよい） |
| データモデル | エンティティごとに見出し + `カラム \| 型 \| 制約 \| 説明` |
| 用語 | `用語 \| 意味` |
| 画面構成 | `画面 / ルート \| 認証 \| 役割`（プロジェクトに応じ列を増減） |

```markdown
| … | … |

|---|---|---|---|
| FR-01 | … | … | … |

## 5. 非機能要件

|---|---|---|---|
| NFR-01 | … | … | … |

## 6. データモデル

| カラム | 型 | 制約 | 説明 |

| … | … | … |

```
